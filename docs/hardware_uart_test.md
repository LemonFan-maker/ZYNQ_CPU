# Hardware UART Test

The ZYNQ_CPU hardware bring-up bitstream is generated by
`vivado/build_hw_bringup.tcl`. The PS UART probe reads a PL AXI-Lite build-id
register at `0x43c00000`, then writes and reads a scratch register to prove the
PS GP0 -> PL path is alive.

Terminal 1:

```sh
scripts/serial_monitor.sh /dev/ttyUSB0 115200
```

If the serial device is different, replace `/dev/ttyUSB0`. If omitted, the
script tries the first `/dev/ttyUSB*` or `/dev/ttyACM*`.

Terminal 2:

```sh
scripts/run_xsct.sh hw_bringup/legacy_gpgpu_uart_test.xsbl
```

Known-good legacy GPGPU test expected first UART lines:

```text
########################################
#  Zynq GPGPU Performance Benchmark    #
########################################
```

For the ZYNQ_CPU bitstream and probe:

```sh
./scripts/run_vivado.sh -mode batch -source vivado/build_hw_bringup.tcl
./scripts/build_ps_uart_probe.sh
./scripts/run_xsct.sh hw_bringup/download_zynq_cpu_bringup.xsbl
```

`scripts/build_ps_uart_probe.sh` first assembles the PL CPU bring-up programs in
`hw_bringup/programs/` and generates
`hw_bringup/build/generated/zx32_programs.h`.
That generated header now contains both raw word arrays and ELF byte blobs.
The custom DataMover runtime test also loads its program from the ELF blob now,
so the UART log is checking the same format you will use for real programs.
There is also an entry smoke test that uses a non-zero `_start` entry point and
verifies the reset-vector path end to end. The supervisor smoke test checks the
first board-level M-mode to S-mode handoff, delegated S-mode `ecall`, and
`sret` return path. The supervisor timer smoke test programs the MMIO timer,
delegates the resulting interrupt to S-mode, and verifies return from the
S-mode interrupt handler. The boot payload smoke test exercises a more
Linux-like handoff: M-mode initializes delegation and enters an S-mode payload
with `a0=hartid` and `a1=dtb`. The supervisor counter smoke test separately
checks `mcounteren` and S-mode reads of `cycle`, `time`, and `instret`.

Expected ZYNQ_CPU UART lines:

```text
########################################
#  ZYNQ_CPU PL bring-up probe          #
########################################
PL build id:  0x26051001
PL status:    0x00000001
Scratch rd:   0x13579bdf
ZYNQ_CPU AXI-Lite probe: PASS
```

The full probe also reports individual PASS/FAIL lines for DataMover loopback,
PL CPU initiated DataMover, BRAM load/run, ELF load/run, entry smoke, trap
smoke, supervisor smoke, supervisor timer smoke, and custom DataMover
instructions. It also reports a boot payload smoke result for the M-mode to
S-mode payload handoff and a supervisor counter smoke result.

The download script uses:

- `build/vivado_hw/zynq_cpu_hw.runs/impl_1/zynq_cpu_system_wrapper.bit`
- `build/vivado_hw/zynq_cpu_hw.gen/.../zynq_cpu_system_processing_system7_0_0/ps7_init.tcl`
- `hw_bringup/build/ps_uart_probe.elf`

The software-side custom instruction interface is defined in
`hw_bringup/zx32_isa.h`, and the matching assembler lives in
`tools/zx32asm.py`.
